# Phase 10 Plan 01: Google Ads Credential Foundation

## Goal

Wire Google Ads authentication into the ops-marketing skill via a 3-tier credential cascade (userConfig → env → Doppler), provide an OAuth2 setup flow in the setup wizard, and add a campaign pre-gather function to the marketing dashboard script. This plan establishes auth only; sub-command implementations follow in Plans 02 and 03.

## Requirements Addressed

- `GADS-07`: Credential resolution and token refresh for Google Ads API access
- Google Ads OAuth2 setup flow integrated into `/ops:setup` wizard

## Runtime Prerequisites

**STOP — Sam must provide the following before this plan can execute:**

| Credential | Where to get it |
|---|---|
| `GADS_DEV_TOKEN` | [Google Ads API Center](https://ads.google.com/aw/apicenter) → Developer token (requires approved API access) |
| `GADS_CLIENT_ID` | Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client ID |
| `GADS_CLIENT_SECRET` | Same OAuth 2.0 Client ID entry in Google Cloud Console |
| `GADS_REFRESH_TOKEN` | Obtained via OAuth2 authorization flow (Plan 01 setup wizard generates this) |
| `GADS_CUSTOMER_ID` | Google Ads account → Settings → Account details (10-digit, format `123-456-7890`) |
| `GADS_LOGIN_CUSTOMER_ID` | Only required for MCC (manager) accounts; the MCC account ID |

**API approval note:** Google Ads API requires applying for access at https://developers.google.com/google-ads/api/docs/access-levels. Basic access is available immediately; Standard access requires application review.

### Task 1: Add Google Ads credential resolution to ops-marketing/SKILL.md

**File**: `claude-ops/skills/ops-marketing/SKILL.md`

Add the following to the Runtime Context section:

```
google_ads_dev_token, google_ads_client_id, google_ads_client_secret,
google_ads_refresh_token, google_ads_customer_id, google_ads_login_customer_id
```

Add `### Google Ads` credential resolution block after the existing credential blocks:

```bash
# 6-variable cascade: preferences.json → env → Doppler
GADS_DEV_TOKEN="${google_ads_dev_token:-${GOOGLE_ADS_DEV_TOKEN:-$(doppler secrets get GOOGLE_ADS_DEV_TOKEN --plain 2>/dev/null)}}"
GADS_CLIENT_ID="${google_ads_client_id:-${GOOGLE_ADS_CLIENT_ID:-}}"
GADS_CLIENT_SECRET=${google_ads_client_secret:-${GOOGLE_ADS_CLIENT_SECRET:-}}
GADS_REFRESH_TOKEN="${google_ads_refresh_token:-${GOOGLE_ADS_REFRESH_TOKEN:-$(doppler secrets get GOOGLE_ADS_REFRESH_TOKEN --plain 2>/dev/null)}}"
GADS_CUSTOMER_ID="${google_ads_customer_id:-${GOOGLE_ADS_CUSTOMER_ID:-}}"
GADS_LOGIN_CUSTOMER_ID="${google_ads_login_customer_id:-${GOOGLE_ADS_LOGIN_CUSTOMER_ID:-}}"
# Strip dashes from customer ID
GADS_CUSTOMER_ID="${GADS_CUSTOMER_ID//-/}"
```

Add access token refresh:

```bash
# Refresh short-lived access token from refresh token
if [ -n "$GADS_REFRESH_TOKEN" ] && [ -n "$GADS_CLIENT_ID" ]; then
  GADS_ACCESS_TOKEN=$(curl -sf -X POST https://oauth2.googleapis.com/token \
    -d "client_id=${GADS_CLIENT_ID}&client_secret=${GADS_CLIENT_SECRET}&refresh_token=${GADS_REFRESH_TOKEN}&grant_type=refresh_token" \
    | jq -r '.access_token // empty')
fi
GADS_HEADERS=(-H "Authorization: Bearer $GADS_ACCESS_TOKEN" -H "developer-token: $GADS_DEV_TOKEN")
# Add login-customer-id header for MCC accounts
[ -n "$GADS_LOGIN_CUSTOMER_ID" ] && GADS_HEADERS+=(-H "login-customer-id: $GADS_LOGIN_CUSTOMER_ID")
```

Add routing table entry: `google-ads, gads` pointing to `## google-ads` section.

Add `## google-ads` section with credential guard and placeholder for Plans 02+03 sub-commands.

Add `google-ads` to `## setup` auto-scan patterns: `GOOGLE_ADS_*` env vars, shell profile grep, Doppler filter, Dashlane, keychain, Chrome history.

## Threat Model

| ID | Threat | Mitigation |
|---|---|---|
| T-10-01-01 | `refresh_token` written to `preferences.json` on disk | Preferences file is `~/.claude/plugins/.../preferences.json`, user-only permissions; document that backup tools should exclude it; never commit the file |
| T-10-01-02 | `localhost:8080` OAuth redirect server left running | Background server receives one request then exits; no persistent listener |
| T-10-01-03 | `access_token` in memory for session duration | Ephemeral, HTTPS-only transport; acceptable for CLI tool context |
| T-10-01-04 | `developer_token` visible in curl command output | Stored in preferences.json / env, never hardcoded in committed files; shell history is user's responsibility |
